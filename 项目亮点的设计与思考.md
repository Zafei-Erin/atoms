# Atoms - 设计说明文档

## 一、项目亮点与扩展性思考

### 1. 两阶段设计流程

这是对atoms最核心的交互设计拓展。不同于直接生成一个完整方案，我将构建过程拆分为两个阶段：

- **Stage 1（提案）**：AI 快速生成 2 个风格不同的设计方向，只做首屏或核心视觉，保持轻量的前端代码，让用户快速预览决策
- **Stage 2（精修）**：用户选择方向后，AI 生成完整、精修的产品原型

**为什么这样设计：**

现在 AI agent 产出项目代码的能力已经很强了，但审美和设计风格往往很单一。真正能让用户眼前一亮、让产品脱颖而出的，很大程度上取决于设计、审美和交互。所以我直接内置了 frontend design 的 skill，让 AI 在生成代码之前先关注设计方向。

在此基础上，用少量的 HTML 先给用户提供设计选项，确认方向后再生成完整项目代码。这样做有两个好处：一是有效减少用户对设计不满意而返工带来的 token 和时间消耗；二是后续生成的项目代码目标更明确，结构更清晰可维护，不会因为反复修改变得混乱。

### 2. Agent Loop

后端的 AI 调用采用 Agent Loop 设计（最多 8 轮），而非单次 LLM 调用。每一轮中 LLM 可以决定调用工具、获取结果、继续推理，直到生成最终回复。工具调用后实时通过 SSE 推送状态（running → complete），前端可以展示调用过程

**为什么用 Agent Loop 而非单次调用：**

相比单次 LLM 调用，Agent Loop 更智能也更灵活。单次调用只能把所有信息一次性塞进 prompt，LLM 没有机会中途获取外部信息或根据中间结果调整策略。而 Agent Loop 让 LLM 可以自主决定下一步做什么——需要什么工具就调用什么工具，拿到结果再继续推理，行为模式更接近真实的开发者。

扩展性上，新增工具只需要定义 schema 和执行函数，注册到 loop 里就行，不需要改动流式传输或前端渲染逻辑。后续加文件操作、依赖安装、终端命令等能力，架构上不需要重构。

模型选择上用的是 MiniMax-M2.7，成本可控，效果好，性价比高。配合 Agent Loop 多轮调用的场景，token 消耗可预期，不会因为模型价格导致成本失控。

### 3. Skills 系统

设计了一个可扩展的 Skills 机制：AI Agent 可以在对话中按需调用 `load_skill` 工具，从文件系统加载 `SKILL.md` 获取专业指令。

**为什么这样设计：**

核心思路是"两层注入，按需加载"。System prompt 里只放 skill 的名称和一句话描述，不放完整内容。这样做有几个好处：

第一，显著缩小 system prompt 的体积。SP 过长会导致模型注意力下降，而且每次请求都要先消耗这些 token，是不必要的开销。同时大多数 provider 会对 system prompt 做缓存，如果把完整 skill 内容塞进 SP，每次加载新技能都会使缓存失效，浪费缓存带来的性能和成本优势。

第二，减少不必要的提示带来的干扰。不是每次对话都需要所有 skill 的完整指令。LLM 根据 skill 描述自行判断是否需要，需要时通过 `load_skill` 工具按需加载完整内容，作为 tool result 注入对话上下文。不需要的 skill 完全不会出现在上下文里，保持对话干净、聚焦。

第三，实现"配置即能力"——扩展 AI 能力只需要添加一个 Markdown 文件，不需要修改代码和重新部署。

## 三、扩展计划与优先级

### 1 - Web Container 集成

当前的 iframe 预览只能渲染单一 HTML 文件，本质上还是一个"看起来能用"的静态展示。下一步计划引入 Web Container 作为运行环境：

- **真正的全栈项目**：支持多文件结构、npm 依赖、后端逻辑，不再局限于纯前端展示
- **沙箱安全**：Web Container 运行在浏览器沙箱内，用户代码不会影响宿主环境，天然隔离
- **用户可参与编写**：不只是 AI 生成代码，用户也可以直接编辑文件、实时看到渲染结果
- **增量更新**：Agent 通过命令操作文件系统，修改哪里更新哪里，不再全量重写

### 2 - Agent 能力增强

计划引入 LangGraph 框架重构 Agent 层，实现更自动化、生产级可用的 Agent 系统。

当前的两阶段设计流程（提案 → 选择 → 精修）是通过 system prompt 约束实现的，能用但不够规范——阶段流转、状态判断都依赖 LLM 对 prompt 指令的遵循，不够可控。用 LangGraph 可以把两阶段流程建模为显式的状态图，每个阶段是一个节点，阶段之间的流转由代码控制而非 prompt 暗示，行为更确定、更可调试。

具体方向：

- **流程图化**：将两阶段设计、多轮迭代等复杂流程用 LangGraph 的状态图显式建模，阶段流转可控可观测
- **自动记忆管理**：利用框架的上下文压缩和记忆管理能力，解决长对话场景下 token 膨胀和注意力下降的问题
- **扩展工具集**：文件读写、依赖安装、终端命令等，让 Agent 具备完整的开发者能力
- **充实 Skills 生态**：提供常见项目模板（SaaS Landing Page、Dashboard、电商首页等），降低生成门槛
- **多模型支持**：Claude、GPT-4o、Zhipu 5.0 等
